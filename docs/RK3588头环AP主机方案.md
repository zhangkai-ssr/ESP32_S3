# RK3588 头环 AP 主机方案

本文根据当前项目讨论内容整理，面向“头环作为 AP 主机，搭载 RK3588，并接入两个 RGB 摄像头”的系统形态。方案重点不是先完成最终 AI 算法，而是先把双手环数据、双相机数据、后续麦克风数据统一接入同一条时间轴，保证后续算法可以按时间戳对齐和回放。

## 1. 方案结论

主机侧建议定义为：

```text
RK3588 头环 AP 主机
  = 2.4GHz Wi-Fi AP
  + 双手环数据接收端
  + 双 RGB 摄像头采集端
  + 统一时间基准
  + 本地存盘与算法输入节点
```

系统允许 Wi-Fi 数据有 10~20 ms 级传输延迟，但每一路数据必须携带时间戳。算法和存盘不应以“数据到达主机的时间”为唯一依据，而应以“采样或曝光发生时的统一时间戳”为依据。

## 2. 系统组成

完整系统包含以下节点：

| 节点 | 角色 | 当前建议 |
|---|---|---|
| 左手环 | EMG / IMU 数据源 | ESP32-S3，连接头环 AP |
| 右手环 | EMG / IMU 数据源 | ESP32-S3，连接头环 AP |
| 头环主机 | AP、收流、同步、存盘、算法运行 | RK3588 |
| RGB 摄像头 0 | 视觉数据源 | USB3.0 或 MIPI CSI |
| RGB 摄像头 1 | 视觉数据源 | USB3.0 或 MIPI CSI |
| 麦克风 | 音频数据源 | 当前预留，后续确认结构与接口 |

推荐拓扑：

```text
左手环 ESP32-S3  STA  \
                    2.4GHz Wi-Fi       RK3588 头环 AP 主机
右手环 ESP32-S3  STA  /                授时 + 收流 + 相机采集 + 存盘 + 算法

RGB Camera 0  ------ USB / MIPI ------/
RGB Camera 1  ------ USB / MIPI -----/
Mic 预留       ------ USB / I2S ------/
```

## 3. 主机硬件建议

### 3.1 RK3588 主控

头环主机建议使用 RK3588 / RK3588S 平台，原因是：

- 有足够 CPU / NPU / 内存资源承载后续 CV / AI 算法。
- 可以同时处理网络收流、双相机采集和本地存盘。
- Linux 生态相对成熟，便于部署采集服务、同步服务和算法服务。

系统建议优先使用：

- Ubuntu 22.04 Server；
- 或板厂提供的 RK3588 Linux SDK。

当前主机是数据汇聚与时间基准，不建议一开始把工作重心放在桌面 UI 上。

### 3.2 Wi-Fi AP

手环侧 ESP32-S3 当前只需要稳定连接 2.4GHz AP，因此主机侧 Wi-Fi 重点是稳定性，而不是追求 Wi-Fi 6 指标。

建议：

- 头环主机开启 2.4GHz AP；
- 使用 WPA2-PSK；
- 固定 SSID / password；
- 固定信道优先，减少运行中信道变化；
- 禁用或降低省电策略对链路抖动的影响；
- 两个手环固定分配或记录 IP / device_id。

Wi-Fi 5 模块已经足够满足当前 ESP32-S3 连接需求，Wi-Fi 6 可作为更高规格选项，但不是当前链路验证的必要条件。

### 3.3 双 RGB 摄像头

摄像头当前建议选型为两个 RGB 摄像头。接口可分两阶段考虑：

第一阶段建议使用 USB 摄像头：

- 调试快；
- Linux 侧 V4L2 / GStreamer / OpenCV 生态成熟；
- 便于快速跑通双路采集、时间戳、存盘和算法接口。

第二阶段再评估 MIPI CSI：

- 集成度更高；
- 延迟和带宽更可控；
- 但硬件、排线、驱动和调试成本更高。

如果后续需要小于 1 ms 的双相机同步精度，摄像头选型时应优先确认是否支持外触发、帧同步或硬件时间戳。

### 3.4 存储

头环主机需要同时写入手环数据和双路视频，建议优先使用：

- NVMe SSD；
- 或高速 eMMC；
- 不建议只依赖普通低速 TF 卡作为主要存储。

存储设计应按连续采集场景评估，而不是只看单次短时间演示。

## 4. 时间同步设计

### 4.1 核心原则

跨设备之间一定存在传输延迟，尤其是 Wi-Fi 链路。因此主机方案的核心不是消灭所有延迟，而是做到：

- 每个数据包知道自己是什么时间采样的；
- 每个相机帧知道自己是什么时间曝光或采集的；
- 主机知道数据是什么时间收到的；
- 后续算法可以根据时间戳把数据放回统一时间轴。

也就是说，手环数据可能在采样后 20 ms 才到达 RK3588，但只要数据包里带有正确时间戳，主机就可以把它放回 20 ms 前的位置。

### 4.2 时间轴定义

建议保留三类时间：

| 时间字段 | 来源 | 用途 |
|---|---|---|
| `device_time_us` | 手环 MCU 或相机设备本地时间 | 原始设备时间 |
| `host_time_us` | RK3588 monotonic 时间轴 | 统一对齐时间 |
| `host_rx_time_us` | RK3588 收到数据时记录 | 评估链路延迟和抖动 |

其中 `host_time_us` 是所有数据融合的主时间轴。

### 4.3 手环时间同步

两个手环与 RK3588 主机之间建议继续使用 UDP 四时间戳同步，逻辑类似 NTP-lite：

```text
T1 = 手环发送 request 时的 MCU 时间
T2 = RK3588 收到 request 时的 host 时间
T3 = RK3588 发送 response 时的 host 时间
T4 = 手环收到 response 时的 MCU 时间
```

计算：

```text
rtt_us    = (T4 - T1) - (T3 - T2)
offset_us = ((T2 - T1) + (T3 - T4)) / 2
```

手环侧可用：

```text
synced_time_us = esp_timer_get_time() + offset_us
```

但建议数据包中同时保留原始 MCU 时间和同步后的估算时间，便于后处理排查。

### 4.4 双相机时间同步

相机同步建议分阶段：

第一阶段，软件时间戳：

- RK3588 每收到一帧，记录 `host_capture_time_us`；
- 如果驱动能提供曝光起始时间，优先记录 `exposure_start_time_us`；
- 每帧记录 `camera_id`、`frame_id`、`host_time_us`。

第二阶段，硬件同步：

- 如果相机支持外触发，用 RK3588 GPIO 或专用同步板给两个相机同时触发；
- 两个摄像头共用 trigger；
- 目标可定义为双相机同步误差小于 1 ms。

### 4.5 麦克风同步

麦克风当前可作为预留项，不阻塞手环和相机主流程。后续加入时建议：

- 记录音频采样起始 `host_time_us`；
- 记录采样率和每个音频块的 sample index；
- 与相机、手环共用同一 session 时间轴。

## 5. 数据包与元数据要求

### 5.1 手环数据包

手环上传 EMG / IMU 时，每包至少应包含：

| 字段 | 说明 |
|---|---|
| `magic` / `version` | 协议识别和版本 |
| `device_id` | 区分左手环 / 右手环 |
| `stream_type` | EMG / IMU |
| `sequence` | 连续性和丢包检查 |
| `device_time_us` | MCU 原始时间 |
| `host_time_us` | 同步到主机时间轴后的估算时间 |
| `sample_count` | 包内样本数 |
| `sample_rate` | 当前采样率 |
| `payload` | 传感器数据 |
| `crc` 或 `footer` | 完整性检查 |

当前 ESP32-S3 项目中已有时间同步和时间戳基础，后续重点是把双设备接收和主机侧统一记录做好。

### 5.2 相机帧元数据

每个相机帧至少记录：

| 字段 | 说明 |
|---|---|
| `camera_id` | 摄像头编号 |
| `frame_id` | 帧序号 |
| `host_time_us` | 主机时间轴上的帧时间 |
| `exposure_time_us` | 曝光时间，能获取则记录 |
| `width` / `height` | 分辨率 |
| `fps` | 采集帧率 |
| `format` | RGB / YUYV / MJPEG / H264 等 |

### 5.3 主机接收记录

RK3588 每收到一包或一帧，都应额外记录：

| 字段 | 说明 |
|---|---|
| `source` | `wrist_left` / `wrist_right` / `cam0` / `cam1` / `mic` |
| `seq_or_frame_id` | 包序号或帧号 |
| `host_rx_time_us` | 主机收到时间 |
| `estimated_delay_us` | 估算延迟 |
| `sync_quality` | 同步状态或质量 |
| `dropped_count` | 丢包或丢帧计数 |

## 6. RK3588 软件服务划分

主机侧建议拆成以下服务：

| 服务 | 职责 |
|---|---|
| `ap_manager` | 启动和维护 2.4GHz Wi-Fi AP |
| `time_sync_server` | 给两个手环提供时间同步 |
| `wristband_receiver` | 接收左 / 右手环 EMG / IMU 数据 |
| `camera_capture` | 同时采集两个 RGB 摄像头 |
| `audio_capture` | 预留麦克风采集 |
| `data_recorder` | 按统一时间轴写盘 |
| `sync_monitor` | 统计 RTT、offset、丢包、抖动 |
| `algorithm_bridge` | 给 CV / AI 算法提供实时或离线数据接口 |

推荐数据流：

```text
手环 / 相机 / 麦克风
      |
      v
采集服务记录原始数据 + 时间戳
      |
      v
统一时间轴缓存
      |
      +--> 本地存盘
      |
      +--> 实时算法
      |
      +--> 离线回放和调试
```

## 7. 存盘格式建议

不建议把所有数据混成单个裸文件。建议每次采集建立一个 session 目录：

```text
session_20260605_001/
  session_meta.json
  wristband_left.bin
  wristband_right.bin
  camera_0.mp4
  camera_1.mp4
  microphone.wav
  timestamps.csv
  sync_log.csv
  receive_stats.json
```

### 7.1 `session_meta.json`

记录本次采集的固定信息：

```json
{
  "session_id": "20260605_001",
  "host": "RK3588 headband AP",
  "wifi_ap": {
    "band": "2.4GHz",
    "ssid": "JSZN_HEADBAND_AP",
    "channel": 6
  },
  "wristbands": [
    {"device_id": 1, "role": "left"},
    {"device_id": 2, "role": "right"}
  ],
  "cameras": [
    {"camera_id": 0, "type": "RGB"},
    {"camera_id": 1, "type": "RGB"}
  ],
  "time_base": "host_monotonic_us"
}
```

### 7.2 `timestamps.csv`

建议字段：

```csv
source,seq_or_frame_id,device_time_us,host_time_us,host_rx_time_us,delay_us,sync_quality
wrist_left,1024,183245678,987654301000,987654321000,20000,locked
cam0,345,0,987654305000,987654305500,500,software_ts
```

### 7.3 `sync_log.csv`

记录同步质量：

```csv
time_us,device_id,rtt_us,offset_us,drift_ppm,valid
987654300000,1,1800,12345678,12.5,true
987654300000,2,2100,12345010,15.1,true
```

## 8. 算法对齐方式

算法侧不应假设所有数据同时到达，而应按照统一时间轴取数据窗口。例如：

- 相机帧时间为 `T`；
- 取左手环 `[T - 10 ms, T + 10 ms]` 的 EMG / IMU；
- 取右手环 `[T - 10 ms, T + 10 ms]` 的 EMG / IMU；
- 如果加入麦克风，取对应音频窗口；
- 将这些数据组成一帧算法输入。

这种方式可以消化 Wi-Fi 延迟和不同设备的到达顺序差异。只要时间戳可信，数据晚到并不等于无法对齐。

## 9. 当前项目需要配合修改的方向

ESP32-S3 手环固件侧后续建议：

1. 每包稳定携带 `device_id`。
2. 时间戳使用 64-bit 微秒时间。
3. 保留 `sequence`，用于判断丢包和乱序。
4. 保留原始 MCU 时间，并提供同步到 host 时间轴的估算值。
5. EMG / IMU 包格式变化时，同步更新主机接收工具。

RK3588 主机侧后续建议：

1. 实现稳定 2.4GHz AP。
2. 实现双手环同时接入和收流。
3. 实现时间同步服务，并记录 RTT / offset / sync_quality。
4. 实现双 RGB 摄像头同时采集。
5. 实现 session 级存盘。
6. 实现离线对齐脚本，用于验证“手环 + 相机”的时间窗匹配。

## 10. 分阶段实施计划

### 阶段一：RK3588 AP 与双手环收流

目标：

- RK3588 开启 2.4GHz AP；
- 两个手环同时连接；
- 主机同时接收两路 EMG / IMU；
- 每包记录 `device_id`、`sequence`、`device_time_us`、`host_rx_time_us`；
- 验证连续收流、吞吐、丢包和乱序。

验收标准：

- 两个手环可持续连接；
- 主机可区分左 / 右手环；
- 每路数据序号连续；
- 能输出基础统计：吞吐、丢包、平均延迟、最大延迟。

### 阶段二：时间同步闭环

目标：

- 手环与 RK3588 建立周期性时间同步；
- 主机记录每个手环的 RTT / offset；
- 数据包可映射到统一 host 时间轴；
- 离线脚本可以根据时间戳重排两路手环数据。

验收标准：

- 双手环数据能在同一时间轴展示；
- Wi-Fi 延迟变化不会破坏样本顺序；
- 能识别同步未锁定、同步质量差、丢包等异常。

### 阶段三：双 RGB 摄像头采集

目标：

- RK3588 同时打开两个 RGB 摄像头；
- 每帧记录 `camera_id`、`frame_id`、`host_time_us`；
- 视频与手环数据写入同一 session；
- 实现相机帧与手环数据的离线时间窗对齐。

验收标准：

- 双路视频可稳定录制；
- 视频帧时间戳连续；
- 可按相机帧时间查到对应左右手环数据窗口。

### 阶段四：硬件同步与算法接入

目标：

- 评估相机外触发或同步线；
- 双相机同步目标小于 1 ms；
- 算法从统一时间轴读取数据；
- 支持实时推理或离线回放。

验收标准：

- 双相机同步误差达到方案目标；
- 算法输入中包含同一时间窗内的相机、左右手环、可选麦克风数据；
- 存盘数据可完整复现实验过程。

## 11. 风险与注意事项

| 风险 | 影响 | 建议 |
|---|---|---|
| Wi-Fi 延迟抖动 | 数据到达顺序不稳定 | 使用时间戳和缓存重排，不以接收顺序作为唯一顺序 |
| 没有统一时间戳 | 后续算法无法可靠对齐 | 所有数据包和视频帧必须记录时间 |
| 双相机不同步 | CV 结果误差变大 | 第一阶段软件时间戳，第二阶段硬件触发 |
| 存储写入不足 | 视频或传感器数据丢失 | 优先 NVMe，持续压测写入带宽 |
| 麦克风方案未确定 | 结构和接口返工 | 当前预留接口，不阻塞主线 |
| 主机服务耦合过重 | 后续调试困难 | AP、同步、收流、相机、存盘分服务实现 |

## 12. 推荐近期工作

近期优先级建议：

1. 确认 RK3588 板卡、Wi-Fi 模块、天线和系统镜像。
2. 跑通 RK3588 2.4GHz AP，确保 ESP32-S3 可稳定连接。
3. 把主机接收工具从单手环扩展为双手环。
4. 将手环数据的 `device_id`、64-bit 时间戳、`sequence` 纳入统一记录。
5. 选定两个 RGB 摄像头，先完成 USB 双路采集验证。
6. 建立 session 存盘目录和 `timestamps.csv`。
7. 写一个离线对齐脚本，验证相机帧 `T` 附近能取到左右手环数据窗口。

## 13. 总结

当前主机方案应围绕“RK3588 头环 AP 作为统一时间和数据中心”展开。硬件上接入两个手环、两个 RGB 摄像头并预留麦克风；软件上先保证 AP、时间同步、双手环收流、双相机采集和统一存盘。

这套方案的关键点是：Wi-Fi 延迟可以存在，但数据不能没有时间戳。只要每包手环数据、每帧相机数据、后续每段音频数据都能落到同一个 host 时间轴上，后续 CV / AI 算法就可以通过缓存和时间窗完成对齐。
