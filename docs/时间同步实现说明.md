# 时间同步实现说明

本文档描述当前 ESP32-S3 固件的时间同步机制，包括板内时间戳打点精度、UDP 在线授时流程、数据包时间戳上传方式，以及已知误差来源与改进建议。

---

## 一、时间轴架构

系统存在两个时钟域：

```
MCU 时钟域                        Host 时钟域
─────────────────────────────     ─────────────────────────────
esp_timer_get_time()              host monotonic_ns / time.monotonic_ns()
  │                               │
  │   UDP 4-timestamp 同步         │
  │   offset_us = f(T1,T2,T3,T4)  │
  │◄──────────────────────────────│
  ▼                               ▼
time_sync_now_us()        host_rx_us（收包时记录）
= esp_timer + offset_us
  │
  ├── EMG 包 timestamp（TCP 3333）
  └── IMU 包 timestamp（TCP 3334）
```

**板内一致性保证**：EMG 和 IMU 使用同一个 `esp_timer` 硬件计数器，应用同一个 `offset_us`，时间轴天然一致，无需额外对齐。

---

## 二、板内时间戳打点精度

| 传感器 | 采样率 | 打点 jitter | 采样周期 | jitter 占比 |
|---|---|---|---|---|
| EMG（ADS1298） | 2000 SPS | **5~15 µs** | 500 µs | 1~3% |
| IMU（LSM9DS1） | 200 Hz | **5~15 µs** | 5000 µs | 0.1~0.3% |

## 三、UDP 在线授时（时间同步协议）

### 3.1 协议

UDP 端口：`3332`

**请求包（18 B）：**

```
magic(4)  ver(1)  type=0x01(1)  device_id(1)  rsvd(1)  seq(2 LE)  t1_us(8 LE)
```

**响应包（34 B）：**

```
magic(4)  ver(1)  type=0x02(1)  device_id(1)  rsvd(1)  seq(2 LE)
t1_us(8 LE)  t2_us(8 LE)  t3_us(8 LE)
```

**四时间戳定义：**

```
T1 = ESP32 发送请求时的 esp_timer 时间（µs）
T2 = Host 收到请求时的 monotonic 时间（µs）
T3 = Host 发送响应时的 monotonic 时间（µs）
T4 = ESP32 收到响应时的 esp_timer 时间（µs）
```

**计算公式：**

```
rtt_us    = (T4 - T1) - (T3 - T2)
offset_us = ((T2 - T1) + (T3 - T4)) / 2
```

### 3.2 同步参数

| 参数 | 数值 | 说明 |
|---|---|---|
| 启动 burst 轮数 | 20 轮 | 取 RTT 最小轮的 offset 作为初始值 |
| 首次 lock 时间 | 1.5~5 s | WiFi 良好约 1.5 s，超时上限 5 s |
| 运行期重同步间隔 | 2000 ms | 每 2 秒更新一次 offset |
| offset 更新方式 | 低通滤波（α=0.15） | 渐进更新，防止 timestamp 突跳 |
| offset 误差上界 | RTT / 2 | 局域网良好时约 0.5~2.5 ms |
| WiFi 断线行为 | 跳过当轮同步 | 保持最后有效 offset 不变 |

## 四、时间戳随数据包上传

时间戳**内嵌在每个数据包中**，随 TCP 数据流同步发送，不存在单独的时间戳上传通道。

### EMG 包（TCP 3333）

```
[0]       0xAA         header
[1]       0x03         version
[2..3]    uint16 LE    seq
[4..11]   int64  LE    mcu_ts_us（time_sync_now_us()，DRDY 时刻）
[12]      uint8        device_count
[13..]    EMG payload
[last]    0x55         footer
```

- 每帧独立携带 64-bit timestamp
- 速率：2000 个包/s

### IMU 包（TCP 3334）

```
[0]       0xBB         header
[1]       0x02         version
[2]       0x20         type
[3..4]    uint16 LE    seq
[5..12]   int64  LE    mcu_ts_us（批次第一个样本的定时器触发时刻）
[13]      uint8        n_samples = 10
[14..]    10 × 20 B    样本数据（int16 LE：ax ay az gx gy gz mx my mz rsvd）
[last]    0x55         footer
```

- 批次首样本 timestamp，10 个样本共享
- 速率：20 个包/s

### Host 侧配合

Host 收到每个包后额外记录 `host_rx_us`，与包内 `mcu_ts_us` 配合做后处理回归：

```
calibrated_time_us = slope × mcu_ts_us + intercept
```

## 五、误差汇总

| 误差来源 | EMG | IMU | 说明 |
|---|---|---|---|
| 打点 jitter | 5~15 µs | 15~60 µs | 硬件事件到 timestamp 的延迟 |
| `time_sync_now_us()` 开销 | < 2 µs | < 2 µs | 可忽略 |
| `esp_timer` 分辨率 | 1 µs | 1 µs | 硬件计数器精度 |
| 晶振短期抖动 | < 1 µs | < 1 µs | 单次采样 |
| 晶振长期漂移 | ~50 µs/s | ~50 µs/s | 由 UDP 同步校正 |
| UDP offset 误差 | 0.5~5 ms | 0.5~5 ms | 含 WiFi RTT，主项 |

**最终 timestamp 精度受限于 UDP offset 误差（0.5~5 ms），内部 jitter 是次要项。**

## 六、当前优化建议

### 6.1 ~~IMU 定时器改为 ISR 派发~~（已完成）

已将 `dispatch_method` 改为 `ESP_TIMER_ISR`，callback 使用 `xSemaphoreGiveFromISR` + `portYIELD_FROM_ISR`。IMU 打点 jitter 已从 15~60 µs 压缩至 5~15 µs，与 EMG 对齐。

### 6.2 缩短重同步周期（提升 offset 跟踪速度）

当前重同步周期为 2000 ms，滤波时间常数约 13 s。建议缩短至 1000 ms，滤波系数从 0.15 调整为 0.3，时间常数缩短至约 3 s，可更快跟踪 WiFi RTT 变化和晶振漂移。

### 6.3 后处理线性回归（消除长期漂移）

Host 侧对每路数据拟合：

```
host_time_us = slope × mcu_ts_us + intercept
```

拟合输入：
- `mcu_ts_us`：包内 timestamp
- `host_rx_us`：Host 收包时的 monotonic 时间

拟合后每个样本的校准时间精度可达 **< 1 ms**，消除晶振漂移的累积影响。

## 七、启动流程中的时间同步顺序

```
nvs_flash_init()
wifi_manager_init()
wifi_manager_wait_connected()
time_sync_start()              ← 启动 UDP 同步任务，开始 20 轮 burst
等待 locked 或超时 5 s         ← main.c 轮询 time_sync_is_locked()
ads1298_init() / stream_start()
imu_stream_init() / start()
```

流数据采集在 time_sync locked 后启动，保证首包 timestamp 已经是 Host 时间轴校正后的值。若 5 s 内未 lock，退回原始 MCU 时间继续推流，Host 侧用 `host_rx_us` 兜底。
