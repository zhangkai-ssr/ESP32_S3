# 协议改动：序列号独立化

> 对象：嵌入式 / 固件同事
> 改动范围：`ble_app.c::pkt_seq` 逻辑（**字节级布局不变**）

---

## 背景

当前固件：EMG 和 IMU 共享同一个全局 `pkt_seq` 计数器，每次 `bt_gatt_notify` 调用 +1。

App 端表现：单路看到的 `msg_num` 序列跳号、步长不固定（EMG 单路可能收到 `0, 2, 3, 5, 6, 7, 9, ...`，因为中间被 IMU 的 Notify 抢走 `pkt_seq` 位置）。

---

## 问题

App 端丢包检测代价不对称：

| 指标 | 当前实现成本 |
|---|---|
| 全局丢包率 | 简单（`pkt_seq` gap 直接算） |
| **EMG 单路丢包率** | **复杂** —— 必须用"预期速率 100 pkt/s × 时间窗"反推 |
| **IMU 单路丢包率** | **复杂** —— 同上，"预期速率 50 pkt/s × 时间窗" |

实战需要三组指标定位问题：

- EMG 单路丢包高 + IMU 0% → 通常指向 ADS1298 SPI 抖动
- IMU 单路丢包高 + EMG 0% → 指向 IMU I2C 或调度问题

当前协议下，单路指标只能用统计学反推，精度差、需要长时间窗（≥ 10 s）、不能逐包判定。

---

## 改动

把 `pkt_seq` 一个全局计数器拆成两个独立计数器：

| 流 | 字节位置 | 含义 |
|---|---|---|
| EMG 包（`0xC101`） | `[1]` | `emg_seq`：仅 EMG 包每次 `bt_gatt_notify` 时 +1 |
| IMU 包（`0xC102`） | `[1]` | `imu_seq`：仅 IMU 包每次 `bt_gatt_notify` 时 +1 |

**字节级布局完全不动**：包大小、帧头帧尾、所有 offset 都不变。仅 `[1]` 字节的语义从"全局序号"变为"流内序号"。

固件改动 < 10 行代码，无外部接口变化，无包结构变化。

---

## App 端收益

```c
gap = current_seq - last_seq - 1;   // 已做 8→32 bit 扩展计数
if (gap > 0) {
    lost_count += gap;
}
```

- 逐包确定，不依赖时间窗
- 不被 BLE jitter 误判
- 三组指标（全局 / EMG / IMU）都能精确算

---

## 失去的性质（可接受）

| 失去 | 评估 |
|---|---|
| "全局严格连续"作为链路健康度便利 | 两路 gap 加和即可恢复全局丢包率 |
| 跨流时间对齐基于 `msg_num` 的共享锚点 | 当前协议本来就不支持（文档明确"不能用 msg_num 跨流对齐"），未被依赖 |
