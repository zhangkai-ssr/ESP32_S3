## 1. 项目定位

当前 ESP32-S3 项目归属为“具身手环”项目，不应和 NRF54L15 “神经腕带”项目混写。

当前 ESP32-S3 项目的主线是：

- 双手佩戴式设备的数据采集与链路验证
- ESP32-S3 侧采集 EMG / IMU 数据
- 通过 2.4G Wi-Fi / TCP 链路传输到 host / dongle / AP 主机
- host 侧进行接收、校验、可视化和后续算法处理

## 2. RK3588 / AP Host 选型结论

历史对话中建议使用 RK3588 级别 AP 主机作为算法和上位机计算平台，其中 Orange Pi 5 Plus 是一个可选方向。

对于 Orange Pi 5 Plus 这类 RK3588 主机，Wi-Fi 模块优先考虑：

```text
RTL8852BE
M.2 E-Key
WiFi6 + BT5.2
2230 规格
双天线
```

选型原因：

- Orange Pi 5 Plus 本身通常不板载 Wi-Fi，需要外接 Wi-Fi 模块或 USB 网卡。
- M.2 E-Key 模块更适合作为固定 AP / host 网络方案。
- 对当前阶段，驱动成熟度和 AP 热点稳定性比理论峰值更重要。

采购或确认时建议问清：

```text
是否适用于 Orange Pi 5 Plus 的 M.2 E-Key 插槽？
是否包含 2 根天线？
是否支持 Linux / Orange Pi OS 下开启 2.4GHz AP 热点？
是否支持 ESP32-S3 稳定连接？
```

## 3. Wi-Fi / AP 使用结论

当前推荐网络结构是：

```text
AP / 热点 / RK3588 host
  ├─ ESP32-S3，作为 Wi-Fi STA 连接 AP
  └─ Host / PC / Dongle，连接同一个 AP
```

只要 ESP32-S3 和 host 在同一个局域网内，就可以通过 IP + TCP / UDP 双向通信。

一般不需要为了“上传”和“下发”开两个 Wi-Fi 热点。一个 Wi-Fi 网络可以承载多条 TCP 业务链路。

推荐端口规划：

```text
3333  EMG 上行        ESP32-S3 -> Host
3334  IMU 上行        ESP32-S3 -> Host
3335  控制/配置下行   Host -> ESP32-S3
```

TCP 本身是双向连接，因此理论上可以在同一个 socket 中同时收发。但当前 EMG 是高频连续数据流，为了调试清晰和稳定性，建议把高频数据流和低频控制命令分端口处理。

## 4. AP 能否开多个热点

一个 AP 是否能开多个热点 / SSID，取决于硬件、驱动和系统：

- 普通路由器或企业 AP：通常支持多个 SSID。
- 手机热点：通常只支持一个 SSID。
- 电脑热点：多数只支持一个，取决于无线网卡和系统驱动。
- ESP32 SoftAP：通常只开一个 SoftAP SSID；可做 AP+STA，但资源和吞吐需要实测。

对当前项目来说，重点不是“开两个热点”，而是：

- ESP32-S3 和 host 是否在同一个稳定局域网内
- EMG / IMU / 控制命令是否通过端口和协议清晰区分
- 长时间传输是否稳定、可检测、可恢复

## 5. 2.4G 抗干扰结论

2.4G 链路不能保证完全不受干扰。当前阶段应把目标定义为：

```text
干扰下可检测、可恢复、少断流、少堆积、可统计
```

关键策略：

- 包内保留 sequence 和 timestamp，用于 host 侧检测乱序、断点、抖动和吞吐。
- ESP32-S3 关闭 Wi-Fi 省电，降低连续推流时的延迟波动。
- 优先测试固定 2.4G 信道，例如 1 / 6 / 11。
- 尽量使用专用 AP，减少其他终端竞争。
- AP / dongle 尽量靠近腕带，减少人体遮挡。
- 真实采样接入后，采样任务和网络发送任务之间建议增加 ring buffer。
- 缓冲区满时要明确策略：实时优先丢旧包，完整性优先阻塞等待，调试优先统计 overflow。

建议统计项：

- 已发送包数
- TCP send 失败次数
- socket 重连次数
- ring buffer overflow 次数
- 最大连续发送阻塞时间
- host 侧序号跳变次数
- 平均 / 峰值吞吐
- 延迟和抖动

## 6. Wi-Fi 信道与省电

如果 ESP32-S3 工作在 STA 模式并连接外部 AP，实际 Wi-Fi 信道由 AP 决定。

可在拿到 IP 后打印当前连接信道：

```c
uint8_t primary;
wifi_second_chan_t second;
esp_wifi_get_channel(&primary, &second);
ESP_LOGI(TAG, "WiFi channel: primary=%u second=%d", primary, second);
```

连续推流场景建议关闭 Wi-Fi 省电：

```c
esp_wifi_set_ps(WIFI_PS_NONE);
```

历史实现中该操作放在 `esp_wifi_start()` 之后、等待联网成功之前。这个时机适合持续 TCP 推流，因为在连接和后续数据发送前已经关闭 modem sleep。

## 7. 当前项目应优先保持的方向

当前项目已经包含多个模块，后续扩展应优先不打断现有数据链路：

- 保持 EMG `3333` TCP 推流稳定
- 保持 IMU `3334` TCP 推流稳定
- 如需下行控制，优先新增 `3335` 控制端口
- 不急于做多客户端广播
- 不急于把所有数据和命令混入同一个 TCP 端口
- 修改协议时，同步更新 host 侧 Python 客户端和可视化工具

建议验证顺序：

1. ESP32-S3 成功连接 AP 并获取 IP。
2. Host 能连接 EMG `3333` 并持续收流。
3. Host 能连接 IMU `3334` 并持续收流。
4. 同时运行 EMG 和 IMU 双路接收。
5. 如新增控制端口，验证 `3335` 下行命令和 ACK。
6. 长时间运行 10 到 30 分钟，记录吞吐、断流、序号连续性、内存和任务栈水位。
